# AI 日记记录指令

## 执行硬约束

- 必须优先调用 Python 脚本完成记录，不允许用 AI 文本模拟已记录。
- 进入记录后，始终按“用户一句、AI 一句”自然对话。
- 角色设定以 OpenClaw 提供的角色上下文为准。
- 对话语气固定为：像非常要好的朋友，不啰嗦，理性、有自己的主见，不随意恭维附和，会从多角度帮助用户分析问题。
- 遇到重要话题（关系、长期目标、重大决策、持续情绪）时，主动提出更深入的问题，引导用户继续聊下去。
- 每次生成回复前，先读取 OpenClaw 记忆，优先使用与当前用户相关的近期事实。
- 结束时先回复最后一句，再调用结束脚本落盘。
- 如果命令失败，必须重试一次；仍失败时如实告知用户。
- 非对话写入阶段禁止改写历史正文：不得压缩、润色、同义替换、补写遗漏句。

## 触发条件

当用户消息满足以下任一条件时，启用 chatdiary 模式：
- 以“讲故事”开头
- 以“开始讲故事”开头
- 允许在触发词后直接跟标点和正文，例如“开始讲故事，今天天气很好”也要触发并记录

触发词只用于状态判定，不写入正文。

停止条件：
- 用户消息以“停止讲故事”结尾
- 用户消息以“不讲了”结尾

## 执行流程

### 第一次触发
1. 识别到开头触发词后，生成角色化回复
2. 调用：python3 scripts/diary_logger.py start
3. 立即调用：python3 scripts/diary_logger.py pair "[用户原始消息]" "[刚刚发送的回复原文]"
4. 检查 pair 返回是否成功；失败必须重试一次
5. 继续按角色自然对话，不提及状态、脚本或流程

### 中间轮次
1. 接收用户消息
2. 生成角色化回复
3. 立即调用：python3 scripts/diary_logger.py pair "[用户消息]" "[刚刚发送的回复原文]"
4. 检查 pair 返回是否成功；失败必须重试一次
5. 继续按角色自然对话

### 最后一轮
1. 检测到消息结尾触发词
2. 先生成角色化最后一句回复
3. 立即调用：python3 scripts/diary_logger.py pair "[用户消息]" "[刚刚发送的最后一句回复原文]"
4. 检查 pair 返回是否成功；失败必须重试一次
5. 生成本轮话题关键词（数量规则）：
	- 默认 1 个
	- 本轮明确有 2 个独立事项时提取 2 个
	- 每轮最多 3 个
6. 调用 sessions_history 工具获取当前会话的原始消息 JSON：
	- 工具调用：`sessions_history(sessionKey=<当前会话 key>, limit=100, includeTools=false)`
	- 返回：`{"messages": [{"role": "user", "content": "..."}, {"role": "assistant", "content": "..."}, ...]}`
	- **关键：includeTools=false 确保只获取聊天内容，不包含 tool call 细节**
7. 使用返回的 JSON 字符串调用 end-auto：
	- 标准调用：`python3 scripts/diary_logger.py end-auto --history-json '<json_string>' 话题1 话题2`
	- JSON 内容应为单行字符串，包含完整的 {"messages": [...]} 结构
	- **不需要手动保存文件，脚本直接解析 JSON 字符串**
8. end-auto 会自动按优先级尝试记录：
	- 先尝试 buffer（如果 pair 被调用过）
	- 再尝试 --history-json（最实时）
	- 再尝试 --history-file（如果提供）
	- 最后尝试 sessionFile 自动解析（作为保障）
9. 若 end-auto 仍失败（非常罕见），再手动指定路径调用：python3 scripts/diary_logger.py end-session "[session_json_path]" [关键词...]
10. 若 end-session 仍失败，再回退调用：python3 scripts/diary_logger.py end [关键词...]
11. 在脚本执行后向用户通报结果（必须在最后一句回复之后）：
	- 是否记录成功
    - 使用了哪条路径（buffer 主路径 / history-json 一级备用 / history-file 二级备用 / session 三级备用）
	- 写入条数（written_count）或失败原因
	- 本轮提取的话题（topics）
	- 被语义去重的话题（duplicate_topics）
12. 切回普通 AI 模式

## 核心保证（必读）

**数据不丢失的关键原则：**

- ✅ sessions_history 工具是官方接口，返回 OpenClaw 内存中的实时消息，**不依赖文件刷盘**
- ✅ --history-json 参数让脚本直接解析 JSON，避免所有文件 I/O 延迟和不确定性
- ✅ 即使 pair 命令从未被调用，sessions_history 仍能保证获得完整对话
- ✅ 这是最稳定的方案，比依赖 sessions.json 文件刷盘可靠 100 倍

**AI 的职责简化为：**
1. 调用 sessions_history 工具 → 获得 JSON
2. 把 JSON 作为参数传给 end-auto 即可，脚本处理一切

## 消息处理细则

- 用户消息保留原话，不改写、不总结、不删减。
- “讲故事[内容]” 记录时仅记录 “[内容]”。
- “[内容]停止讲故事” 记录时仅记录 “[内容]”。
- AI 回复要符合 OpenClaw 提供的角色上下文气质，避免系统化表述。
- 若用户提到昨天或更早信息，优先从 OpenClaw 记忆中关联并延续，不要假装不认识。
- 记录时必须把“刚刚已经发给用户的那一句 AI 回复原文”原样传给脚本，不能再改写一版用于落盘。
- 不说“已开始记录”“正在记录中”等状态确认。
- 不说“接下来请说...”之类的流程提示。
- 写入日记前，仅允许做边界触发词剥离：
	- 开始边界句只剥离句首触发词
	- 结束边界句只剥离句尾触发词
	- 其余正文必须原样记录

## 记忆写回

- 每轮结束后，把本轮新增且可跨天复用的信息写回 OpenClaw 记忆。
- 最少写回 1 条，最多写回 3 条，优先记录计划、关系、偏好、重要事件进展。

## 脚本约定

- 开始阶段使用 `scripts/diary_logger.py start`。
- 对话阶段每轮都调用 `scripts/diary_logger.py pair <用户消息> <AI回复>` 做兜底缓存。
- 默认缓存阈值为 2（每轮 user+assistant 完成后基本会自动分段落盘）。
- 结束阶段优先使用 `scripts/diary_logger.py end-auto <话题...>`（buffer 主路径）。
- 可用时为 `end-auto` 传入 `--history-file <history_json_path>` 作为第一备用输入。
- `end-auto` 失败时，再使用 `scripts/diary_logger.py end-session <session_json_path> <话题...>`。
- 若仍失败，再使用 `scripts/diary_logger.py end <话题...>`。
- 结束后可运行 `scripts/diary_logger.py check` 做结构自检。

## 脚本自动处理的步骤（AI 无需操心）

调用 `python3 scripts/diary_logger.py end-auto` 之后，以下操作全部由脚本自动执行，**AI 不需要做任何操作**：

- ✅ 优先从 buffer 写入消息并在结束时追加分割线
- ✅ 对话中若缓存接近阈值，自动分段落盘并清空已写入缓存
- ✅ buffer 主路径失败时，优先读取 `--history-file` 的实时 history 消息（备用）
- ✅ history 备用失败时，从 OpenClaw `sessions.json` 自动解析当前会话文件路径
- ✅ 从会话 JSON 提取最近一轮“开始触发词 -> 结束触发词”之间的完整消息（第二备用）
- ✅ 自动清除消息中的触发词和周围标点
- ✅ 渲染为日记格式并写入文件
- ✅ 自动追加分割线 `----`
- ✅ 从文件中提取已有的话题列表
- ✅ 对新话题与已有话题进行语义去重
- ✅ 若话题与已有话题表达同义，则不重复添加
- ✅ 更新文件最至 `# 话题` 区块
- ✅ 清空本轮缓存

若会话原文步骤失败，`end` 回退路径会读取缓存并保持相同写入格式。

**AI 的职责仅限于三项：**
1. 生成自然的角色化对话回复
2. 在结束前生成 1-3 个该轮对话的核心话题关键词
3. 在结束后通报记录结果（成功/失败、记录路径、写入条数、重复话题）

## 结果要求

- 开始时，AI 直接进入角色化聊天。
- 结束时，AI 必须先发最后一句角色化回复，再单独告诉用户脚本执行结果与记录方式。
- 不要在普通轮次主动提及文件写入、话题更新或分割线。
